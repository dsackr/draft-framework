# Workspaces And The DRAFT App

## Operating Model

DRAFT separates the public framework from private company content.

The public framework repo owns:

- `app/` for the shared UI and API
- `framework/schemas/` for object contracts
- `framework/configurations/` for base ODCs, compliance profiles, control
  catalogs, and domains
- `framework/tools/` for validation and generation
- `examples/catalog/` for demonstration content only

A company private workspace repo owns:

- `catalog/` for architecture content
- `configurations/` for company overlays and extensions
- `.draft/workspace.yaml` for tracked workspace settings
- `.draft/framework.lock` for the pinned framework version

The DRAFT App resolves the effective model in this order:

1. framework base configuration
2. workspace configuration overlays
3. workspace catalog content

Company overrides should be represented as `object_patch` YAML generated by the
app/API. Patch files live under `configurations/object-patches/`. A patch
targets an existing object and deep-merges selected fields without copying the
full base object into the private workspace.

## AI Agent Contract

AI agents should use the same API as the UI when the app is available. The
minimum useful flow is:

1. Load `/api/framework/manifest`.
2. Load `/api/workspace/effective-model`.
3. Compare source material against existing effective objects.
4. Create or update objects through `/api/objects/upsert`.
5. Create overrides through `/api/object-patches/upsert`.
6. Run `/api/workspace/validate`.
7. Commit through `/api/git/commit`.
8. Publish through `/api/publish`.

Direct YAML editing is acceptable for framework development and recovery, but
company authoring should prefer the app/API so overrides, validation, and Git
workflow rules stay consistent.

## Git Workflow

The default workspace workflow is:

- develop changes on the configured dev branch
- commit catalog and configuration updates there
- publish to production by opening a pull request to the configured production
  branch

Small teams may configure direct publishing, but pull requests are the default
because architecture changes are governance artifacts.

## Framework Updates

Framework updates are pinned and explicit. A workspace records the selected
framework reference in `.draft/framework.lock`. The app may detect that a newer
framework version is available, but it should apply updates only after the user
chooses to update and validation passes against the effective workspace model.

## Deployable Architecture Direction

The end goal is deployable architecture. Catalog objects should capture facts
that can later inform pipeline and infrastructure automation:

- RBBs describe reusable deployable building blocks.
- RAs describe deployable patterns.
- SDMs describe product deployment reality.
- Compliance profiles describe required controls.
- Future automation mappings will translate approved objects into pipeline and
  IaC inputs.

When an AI agent has a choice between vague narrative and structured deployable
facts, it should choose structured deployable facts and preserve missing details
as validation gaps or drafting-session questions.
